# Renderer v0.3 — Job System & Thread-Local Arenas — Design Spec

**Date:** 2026-07-01
**Status:** Approved for planning
**Scope:** Engine v0.3 — a general-purpose `JobSystem` (fixed thread pool), per-worker
thread-local command buffers, parallel render-command generation with a merge
step, demonstrated by an animated swarm of thousands of mesh "bots".
**Builds on:** v0.2 mesh path + sort keys + `ResourceRegistry` (merged to `main`).

## 1. Goal & Motivation

v0.1/v0.2 generate all render commands on the main thread: `BotArenaGame::onRender`
submits into a single `Arena`-backed `RenderQueue`, which is sorted and executed
by the backend. That is the exact seam Molecule-style rendering is designed to
parallelize — command *generation* (visibility, transforms, sort-key creation)
is embarrassingly parallel; only GL execution must stay single-threaded.

v0.3 realizes that split:

1. **`JobSystem`** — a reusable fixed thread pool with `parallelFor` and stable
   per-lane worker indices. Owned by `Application`; AI and physics reuse it later.
2. **Thread-local command buffers** — each worker lane owns its own `Arena` +
   `RenderQueue`, so workers allocate and record commands with no locks and no
   false sharing.
3. **Merge + sort** — after generation, the per-lane buffers are concatenated and
   `std::stable_sort`ed, then executed on the main thread.

It is demonstrated by an **animated bot swarm**: thousands of small lit cubes whose
`DrawMesh` commands are produced in parallel via `parallelFor`. This makes the
job system a real workload and matches the project's thousands-of-autonomous-bots
goal. Workers never touch OpenGL.

## 3. Decisions (locked)

| Decision | Choice |
| --- | --- |
| Demo | Animated swarm of thousands of mesh "bots" (each an individual draw call; no instancing yet) |
| JobSystem home | `engine/core/JobSystem`, owned by `Application`, passed to `Renderer` |
| Worker lanes | `workerCount() = workerThreads + 1` (main thread participates as lane 0) |
| Default threads | `hardware_concurrency() - 1` (min 0 → serial fallback) |
| Thread-local buffers | one `{Arena, RenderQueue}` per lane, owned by `Renderer`, reset per frame |
| Parallel API | `Renderer::generateMeshes(count, fn)` over `JobSystem::parallelFor` |
| Merge | concatenate lanes in index order, `std::stable_sort` by the v0.2 sort key |
| Determinism | opaque + depth-tested draws → image invariant to parallel scheduling |
| Bot count | configurable `kBotCount`, default 2048 |
| GL threading | unchanged — execute runs on the main thread; workers never call GL |
| DoD | screenshot renders the full swarm without crashing; job-system units pass |

## 4. Architecture & data flow

```
Application
  ├── owns JobSystem (fixed thread pool)
  └── owns Renderer(JobSystem&)
              │  CommandBufferPool: lanes[0..workerCount-1] = {Arena, RenderQueue}
              ▼
BotArenaGame::onRender(renderer, w, h)
  │  (once) register materials + read cube-mesh handle
  │  renderer.setViewProjection(camera.viewProjection())
  │  serial:  grid + wall meshes  →  renderer.queue()   (lane 0)
  │  parallel: renderer.generateMeshes(kBotCount, batchFn)
  │              → JobSystem::parallelFor(kBotCount, kBatch, task)
  │                task(begin,end,worker): batchFn(lanes[worker].queue, begin, end)
  │                  → MeshRenderer(queue, registry, camera).submit(...) per bot
  ▼   (Renderer::endFrame)
  merge: for lane in lanes: append lane.queue.entries() → m_merged
  std::stable_sort(m_merged) by [layer|shader|material|depth]
  backend.execute(m_merged, viewProjection, laneArena0, registry)
  ▼   next Renderer::beginFrame: reset every lane's arena + clear its queue
GPU
```

## 5. JobSystem — `engine/core/JobSystem.hpp` (+ `.cpp`)

```cpp
class JobSystem {
 public:
  explicit JobSystem(unsigned workerThreads = 0);  // 0 => hardware_concurrency()-1
  ~JobSystem();                                     // stop + join

  JobSystem(const JobSystem&) = delete;
  JobSystem& operator=(const JobSystem&) = delete;

  // Lanes including the main/dispatch lane. Always >= 1.
  size_t workerCount() const { return m_workerCount; }

  // Split [0,count) into ceil(count/batchSize) batches. Each batch runs as
  // fn(begin, end, lane) on some lane in [0, workerCount()). Blocks until all
  // batches finish; the calling (main) thread also drains batches as lane 0.
  void parallelFor(size_t count, size_t batchSize,
                   const std::function<void(size_t begin, size_t end,
                                            size_t lane)>& fn);

 private:
  // thread pool internals: worker threads, a batch queue, mutex + condvar,
  // a per-run completion counter. Workers use lanes 1..N; main uses lane 0.
};
```

Behavior:
- Construction spawns `workerThreads` threads (default `max(0, hw-1)`); each loops
  waiting for batches. `workerCount()` is `workerThreads + 1`.
- `parallelFor` enqueues all batches under the lock, notifies workers, then the
  main thread pops-and-runs batches itself (as lane 0) until the queue is empty,
  and finally waits on the completion counter so all in-flight worker batches
  finish before returning.
- Serial fallback: `workerThreads == 0` → `workerCount() == 1`, and `parallelFor`
  runs every batch on lane 0 (main). Deterministic, headless-safe, and what CI /
  single-core machines get.
- `count == 0` returns immediately.

Only pure command generation runs inside `parallelFor`; nothing calls OpenGL.

## 6. Thread-local command buffers & merge (Renderer)

```cpp
struct WorkerBuffer {
  explicit WorkerBuffer(std::size_t bytes) : arena(bytes), queue(arena) {}
  Arena arena;
  RenderQueue queue;
};
```

`Renderer`:
- Constructor takes `JobSystem& jobs`, stores the reference, and builds
  `m_lanes` — a `std::vector<Scope<WorkerBuffer>>` of size `jobs.workerCount()`
  (heap-boxed because `Arena` is non-movable), each sized to a per-lane budget
  (e.g. 8 MiB / lane).
- `queue()` returns `m_lanes[0]->queue` (the main lane) for serial producers
  (`DebugRenderer`, wall meshes).
- `registry()` / `unitCubeMesh()` / `meshShader()` unchanged (v0.2).
- New: `void generateMeshes(size_t count, const std::function<void(RenderQueue&,
  size_t begin, size_t end)>& fn)` →
  `m_jobs.parallelFor(count, kBatch, [&](b,e,lane){ fn(m_lanes[lane]->queue, b, e); })`.
  `kBatch` is a fixed reasonable batch size (e.g. 128).
- `beginFrame`: for each lane, `arena.reset()` + `queue.clear()`; `backend->beginFrame`.
- `endFrame`: clear `m_merged`; for each lane append `queue.entries()`;
  `std::stable_sort(m_merged, byKey)`; `backend->execute(m_merged, m_viewProjection,
  m_lanes[0]->arena, m_registry)`; `backend->endFrame()`.

`RenderQueue::entries()` already exposes the per-lane `RenderEntry` list. The
merge is a plain concatenation — extract it as a free/static helper
`mergeLaneEntries(lanes, out)` so it can be unit tested without GL.

Pointer lifetime: each `RenderEntry::command` points into its lane's arena, which
lives until the next `beginFrame`; the merged list and its `execute` all happen
in the same frame before any reset. Safe.

## 7. The swarm (`BotArenaGame`)

- `static constexpr size_t kBotCount = 2048;` (configurable constant).
- Each bot's world transform is a pure function of `(index, m_time)` — e.g. a
  position on an animated ring/lattice around the arena centre plus a small bob —
  so generation needs no shared per-bot state and is trivially parallel.
- A small fixed set of materials (e.g. 3 colours chosen by `index % 3`), all using
  the mesh shader; registered once on the first frame (main thread) before any
  `generateMeshes` call, so the registry is read-only during parallel generation.
- `onRender`:
  1. once: register the 3 bot materials (+ keep the v0.2 wall/obstacle materials
     for the walls).
  2. `renderer.setViewProjection(camera.viewProjection())`.
  3. serial: `DebugRenderer` grid; wall meshes via a main-lane `MeshRenderer`.
  4. parallel: `renderer.generateMeshes(kBotCount, [&](RenderQueue& q, size_t b,
     size_t e){ MeshRenderer mesh(q, renderer.registry(), *camera); for (i in
     [b,e)) mesh.submit(cube, botMaterial(i), botTransform(i, m_time)); });`

The arena walls and grid remain as spatial context; the swarm fills the interior.

## 8. Application wiring

`Application` owns `Scope<JobSystem> m_jobs`, constructed before the `Renderer`,
and passes it: `m_renderer = std::make_unique<Renderer>(*m_jobs)`. The render loop
is unchanged (`beginFrame` → layers `onRender` → `endFrame` → optional screenshot
→ swap). The `JobSystem` outlives the `Renderer` (declare `m_jobs` before
`m_renderer` so destruction order tears the renderer down first).

## 9. Testing

- **Unit (Catch2, no GL):**
  - `parallelFor` visits every index exactly once: an `std::atomic<size_t>` counter
    equals `count`, and an atomic sum of indices equals `count*(count-1)/2`.
  - Lane indices are always `< workerCount()`.
  - `count == 0` is a no-op; `workerThreads == 0` (serial) still visits everything.
  - Race/coverage: `parallelFor` writing each index into `results[index]=index`
    across a large `count` yields no lost or duplicated entries (a data race would
    drop or double writes).
  - `mergeLaneEntries`: concatenates lane entry lists in order and preserves every
    entry (built from plain `RenderEntry` values — no GL).
- **Behavioral:** `BOTARENA_SCREENSHOT` renders the full swarm (thousands of lit
  cubes inside the arena) without crashing; visually confirmed. Document a
  ThreadSanitizer test run (`-fsanitize=thread`) as the race check for the job
  system.

## 10. Out of scope (later milestones)

Work-stealing, job priorities, job dependencies/futures, double-buffered render
world (rendering one frame behind simulation), GPU instancing (each bot stays an
individual draw call this milestone), transparency ordering, and AI/physics
actually consuming the `JobSystem`. The `JobSystem` API and the per-lane buffer
model are designed to accept these without changing the public rendering surface.
