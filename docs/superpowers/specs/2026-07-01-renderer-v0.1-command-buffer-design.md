# Renderer v0.1 — Stateless Command-Buffer — Design Spec

**Date:** 2026-07-01
**Status:** Approved for planning
**Scope:** Engine v0.1 — refactor the immediate renderer into a stateless,
layered, command-buffer architecture with a per-frame arena allocator, an
abstract render backend, and an OpenGL backend. Same visible scene.
**Inspiration:** Stefan Reinalter, "Stateless, Layered, Multi-Threaded
Rendering" (Molecule Engine). Ideas adopted: stateless render commands, render
layers, sort keys, backend abstraction, linear/arena allocators.

## 1. Goal & Motivation

Replace the current immediate-mode renderer (`renderer.drawGrid/drawCube/
drawLine`, where each call mutates hidden GL state) with a **stateless
command-buffer** renderer: game code submits self-contained `RenderCommand`s to
a `RenderQueue`; the queue is sorted by a 64-bit key; a `RenderBackend` executes
the sorted commands. Nobody outside the backend touches OpenGL.

This is the engine's public rendering API for the long term. It is the first of
several milestones that add capability without changing that API (see §2). v0.1
proves the pipeline end-to-end while producing the exact same visible scene, so
the refactor is verifiable by screenshot regression.

Two constraints shape v0.1:

1. **Use the arena (linear/bump) allocator primitive.** Commands and generated
   vertex data are allocated from a per-frame arena that resets each frame. The
   arena is a small, unit-tested primitive and the seed for the thread-local
   arenas introduced in a later milestone.
2. **Backend independence.** Command execution goes through an abstract
   `RenderBackend`; the OpenGL implementation is one backend behind that seam
   (Vulkan later).

## 2. Roadmap context (this spec is v0.1 only)

The stateless command buffer is the public API; each milestone layers capability:

| Milestone | Adds |
| --- | --- |
| **v0.1 (this spec)** | Command buffer, render layers, sort keys, per-frame `Arena`, `DebugRenderer`, abstract `RenderBackend` + `OpenGLBackend`. Same scene. |
| v0.2 | Real sort keys (material/depth), filled Opaque geometry, RHI resources (VertexArray/Shader/Texture) as backend building blocks. |
| v0.3 | JobSystem + thread-local arenas + parallel command generation, then merge. |
| v0.4 | Frame resources, framebuffers, render passes (RHI FrameBuffer/RenderPass). |
| v0.5 | Render graph (passes as nodes with dependencies). |
| later | Vulkan backend; GPU-driven culling / indirect draws. |

The previously written RHI foundation spec/plan is not discarded: its resource
abstractions become the backend's building blocks from v0.2 onward.

## 3. Decisions (locked)

| Decision | Choice |
| --- | --- |
| Relationship to RHI | Command layer on top; RHI becomes the backend's toolbox, pulled in from v0.2 |
| Command storage | Flat POD `RenderCommand`, bump-copied into a per-frame `Arena`; `RenderQueue` holds `{sortKey, RenderCommand*}` |
| Memory primitive | `Arena` linear allocator, `reset()` per frame; commands + aux line-vertex data allocated from it |
| Namespace / paths | Existing `engine::` namespace, `engine/renderer/` + `engine/core/` (NOT `Engine::` / `src/Engine/`) |
| Visible scope / DoD | Identical-scene refactor; screenshot must match today's arena scene |
| Camera | `Renderer::setViewProjection(mat4)` set per frame by the game (not a `SetCamera` command yet) |
| Command color | `glm::vec4` (alpha = 1 for now) |
| Sort key | 64-bit `[layer:8][sequence:56]`; submission-order tiebreak → deterministic output |
| Backend seam | Abstract `RenderBackend`; `OpenGLBackend` is the only implementation in v0.1 |
| Old design | Immediate `Renderer` interface + `OpenGLRenderer` deleted |
| Tests | Catch2 (new dep) for `Arena`, sort-key, `RenderQueue::sort`; screenshot for behavior |

## 4. Architecture & data flow

```
BotArenaGame::onRender(renderer, w, h)
   │  renderer.setViewProjection(camera.viewProjection())
   │  DebugRenderer debug(renderer.queue());
   │  debug.drawGrid(...); debug.drawCube(...); debug.drawLine(...)
   ▼
RenderQueue::submit(cmd)
   │     - bump-copy the POD command into the frame Arena
   │     - append {sortKey, RenderCommand*} to the entry list
   ▼   (Application: renderer.endFrame())
RenderQueue::sort()
   │     - std::sort on the 64-bit key: [layer:8][sequence:56]
   ▼
RenderBackend::execute(entries, viewProjection)
   ▼
OpenGLBackend
   │     - expand each command to line vertices INTO the arena
   │       (Cube -> 24 wireframe edge verts, Line -> 2 verts)
   │     - upload once to a dynamic VBO, set u_viewProjection
   │     - glDrawArrays(GL_LINES, ...)
   ▼   next frame: Arena::reset(); RenderQueue::clear()
GPU
```

Frame boundary (`Application::run` loop):

```
renderer.beginFrame(w, h)   // arena.reset(); queue.clear(); backend.beginFrame -> viewport + glClear
for layer : layers          // layer->onRender submits commands
renderer.endFrame()         // queue.sort(); backend.execute(entries, viewProjection)
[screenshot? saveScreenshot; break]
window.swapBuffers()
```

## 5. Components

All under `namespace engine`.

### 5.1 Arena — `engine/core/Arena.hpp` (+ `.cpp`)

A linear/bump allocator over a single heap block.

```cpp
class Arena {
 public:
  explicit Arena(std::size_t capacityBytes);
  ~Arena();
  Arena(const Arena&) = delete;
  Arena& operator=(const Arena&) = delete;

  void* allocate(std::size_t size, std::size_t alignment = alignof(std::max_align_t));

  template <typename T>
  T* alloc(std::size_t count = 1) {
    return static_cast<T*>(allocate(sizeof(T) * count, alignof(T)));
  }

  void reset();                       // rewinds offset to 0; does not free
  std::size_t used() const;
  std::size_t capacity() const;

 private:
  std::byte* m_base = nullptr;
  std::size_t m_capacity = 0;
  std::size_t m_offset = 0;
};
```

- `allocate` aligns `m_offset` up to `alignment`, returns the pointer, advances.
- On overflow (aligned request exceeds capacity) it throws
  `std::bad_alloc`-style `std::runtime_error("Arena: out of memory")`.
- `reset` sets `m_offset = 0` (bump reuse; no destructors run — POD only).

### 5.2 RenderCommand — `engine/renderer/RenderCommand.hpp`

```cpp
enum class RenderLayer : uint8_t { Grid = 0, Opaque = 1, Debug = 2, UI = 3 };
enum class RenderCommandType : uint8_t { Line, Cube };

struct RenderCommand {
  RenderCommandType type = RenderCommandType::Line;
  RenderLayer layer = RenderLayer::Debug;

  glm::vec3 position{0.0f};
  glm::vec3 scale{1.0f};
  glm::vec4 color{1.0f};

  glm::vec3 lineStart{0.0f};
  glm::vec3 lineEnd{0.0f};

  uint64_t sortKey = 0;
};

inline uint64_t makeSortKey(RenderLayer layer, uint64_t sequence) {
  return (static_cast<uint64_t>(layer) << 56) | (sequence & 0x00FFFFFFFFFFFFFFull);
}
```

`RenderCommand` is a POD (trivially copyable), safe to bump-copy into the arena.

### 5.3 RenderQueue — `engine/renderer/RenderQueue.hpp` (+ `.cpp`)

```cpp
struct RenderEntry {
  uint64_t sortKey;
  const RenderCommand* command;
};

class RenderQueue {
 public:
  explicit RenderQueue(Arena& arena);

  void submit(const RenderCommand& command);   // copies into arena, records entry
  void clear();                                 // clears entry list (arena reset elsewhere)
  void sort();                                  // std::sort by sortKey
  const std::vector<RenderEntry>& entries() const;

 private:
  Arena& m_arena;
  std::vector<RenderEntry> m_entries;
  uint64_t m_sequence = 0;                      // monotonic tiebreak within a frame
};
```

- `submit` allocates a `RenderCommand` in the arena, copies the argument in,
  overwrites its `sortKey` with `makeSortKey(command.layer, m_sequence++)`, and
  appends `{sortKey, ptr}`.
- `clear` empties `m_entries` and resets `m_sequence = 0`.
- The entry list is a `std::vector` in v0.1 (can move into the arena later).

### 5.4 RenderBackend — `engine/renderer/RenderBackend.hpp`

```cpp
class RenderBackend {
 public:
  virtual ~RenderBackend() = default;
  virtual void beginFrame(int width, int height) = 0;   // viewport + clear + depth state
  virtual void execute(const std::vector<RenderEntry>& entries,
                       const glm::mat4& viewProjection, Arena& scratch) = 0;
  virtual void endFrame() = 0;
  virtual void readPixels(int x, int y, int width, int height, void* out) = 0;

  static Scope<RenderBackend> Create();          // switches on a backend enum (OpenGL for now)
};
```

`execute` receives the frame arena as `scratch` for building vertex data.

### 5.5 OpenGLBackend — `engine/renderer/opengl/OpenGLBackend.hpp` (+ `.cpp`)

- Owns the line shader (position `vec3` + color `vec4`, uniform
  `u_viewProjection`) and one dynamic VAO/VBO — reusing the current
  `OpenGLRenderer` GL logic (`compileShader`, `createShaderProgram`, the VBO
  layout, `glDrawArrays(GL_LINES)`).
- `beginFrame`: `glViewport`, `glEnable(GL_DEPTH_TEST)`, `glClearColor` +
  `glClear`.
- `execute`: iterate `entries`; for each command expand to line vertices into
  `scratch` (Cube → 12 edges = 24 vertices using the current 8-corner/edge
  tables; Line → 2 vertices), accumulate into one contiguous array, upload with
  `glBufferSubData`, set `u_viewProjection`, `glDrawArrays(GL_LINES, 0, count)`.
- `readPixels`: `glReadPixels(GL_RGB, GL_UNSIGNED_BYTE)` (moved from the old
  renderer's screenshot code).

### 5.6 Renderer — `engine/renderer/Renderer.hpp` (+ `.cpp`, replaces old)

```cpp
class Renderer {
 public:
  Renderer();                    // constructs Arena, RenderQueue, Backend
  void beginFrame(int width, int height);
  void endFrame();
  void setViewProjection(const glm::mat4& viewProjection);
  RenderQueue& queue();
  void saveScreenshot(const std::string& path, int width, int height);

 private:
  Arena m_arena;                 // sized generously, e.g. 8 MiB
  RenderQueue m_queue;
  Scope<RenderBackend> m_backend;
  glm::mat4 m_viewProjection{1.0f};
  int m_width = 0, m_height = 0;
};
```

- `beginFrame`: `m_arena.reset(); m_queue.clear(); m_backend->beginFrame(w,h)`;
  stores `w/h`.
- `endFrame`: `m_queue.sort(); m_backend->execute(m_queue.entries(),
  m_viewProjection, m_arena); m_backend->endFrame()`.
- `saveScreenshot`: `m_backend->readPixels(...)` + `stbi_write_png` (flip
  vertically), same behavior as today.
- Instance owned by `Application`, passed to `Layer::onRender(Renderer&, w, h)`
  (signature unchanged).

### 5.7 DebugRenderer — `engine/renderer/DebugRenderer.hpp` (+ `.cpp`)

Thin producer over a `RenderQueue&` (fills the currently-empty stub):

```cpp
class DebugRenderer {
 public:
  explicit DebugRenderer(RenderQueue& queue);
  void drawLine(const glm::vec3& a, const glm::vec3& b, const glm::vec4& color);
  void drawCube(const glm::vec3& center, const glm::vec3& size,
                const glm::vec4& color);
  void drawGrid(float halfSize, float spacing, const glm::vec4& color);
 private:
  RenderQueue& m_queue;
};
```

- `drawLine` → `RenderCommand{type=Line, layer=Debug, lineStart=a, lineEnd=b, color}`.
- `drawCube` → `RenderCommand{type=Cube, layer=Opaque, position=center, scale=size, color}`.
- `drawGrid` → one `Line` command per grid line, `layer=Grid`.

## 6. Layer → draw-order mapping (reproduces current output)

Current draw order is grid, then cubes, then the yellow bot line. Sort keys make
this deterministic: `Grid(0) < Opaque(1) < Debug(2) < UI(3)`, and within a layer
the submission-order sequence preserves order. Depth testing is on, so the
resulting image matches today's scene exactly.

| Current call | v0.1 command | Layer |
| --- | --- | --- |
| `drawGrid(10, 1, grey)` | N × `Line` | Grid |
| `drawCube(...)` (walls, obstacles, bot) | `Cube` | Opaque |
| `drawLine(bot, origin, yellow)` | `Line` | Debug |

## 7. Removed / changed files

- **Delete:** `engine/renderer/opengl/OpenGLRenderer.{hpp,cpp}`; the immediate
  `engine/renderer/Renderer.hpp` interface (rewritten as the concrete Renderer).
- **Change:** `engine/core/Application.{hpp,cpp}` — own the new `Renderer`,
  loop uses `beginFrame/endFrame`. `game/BotArenaGame.cpp` — `onRender` picks
  the active camera from the existing controllers, calls
  `renderer.setViewProjection(camera.viewProjection())`, then constructs a
  `DebugRenderer(renderer.queue())` and submits grid/cubes/lines through it
  (replacing today's `renderer.setCamera` + `drawGrid/drawCube/drawLine`).
  `game/BotArenaGame.hpp` is essentially unchanged (camera controllers, Input,
  and mode-cycling all stay).
- **Unchanged:** `Camera` + controllers, `Input`, `Time`, `Window`,
  `GraphicsContext`, SDL platform.

## 8. Testing

- **Unit (Catch2, no GL context):**
  - `Arena`: sequential allocations bump the offset; alignment rounds up
    correctly; `reset()` returns `used()` to 0 and reuses memory; `used()` /
    `capacity()` report correctly; an over-capacity request throws.
  - `makeSortKey`: layer occupies the high 8 bits; sequence the low 56;
    `Grid` key < `Opaque` key < `Debug` key for equal sequence; within a layer,
    higher sequence sorts later.
  - `RenderQueue::sort`: submitting Debug, Grid, Opaque in that order yields
    entries ordered Grid, Opaque, Debug; two same-layer submissions keep
    submission order.
- **Behavioral:** `BOTARENA_SCREENSHOT=out.png ./bot_arena` renders the arena
  scene (grid, wall cubes, blue obstacles, red bot + yellow line) identically to
  the current build. Regression = matching image.

## 9. Out of scope (later milestones)

Real sort keys (material/mesh/depth), filled/shaded geometry, RHI resource
classes (VertexArray/Shader/Texture/Framebuffer), textures, render passes,
job-system multithreading, thread-local arenas + buffer merging, type-erased
command packets, render graph, Vulkan backend. The command-buffer API and the
`RenderBackend` seam are designed to accept these without changing the public
surface.
