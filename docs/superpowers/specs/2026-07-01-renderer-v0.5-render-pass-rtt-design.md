# Renderer v0.5 — Render Pass & Render-to-Texture — Design Spec

**Date:** 2026-07-01
**Status:** Approved for planning
**Scope:** Engine v0.5 — extract a `RenderPass{target, clear}` value, lift pass
orchestration from the backend into the `Renderer`, and add a picture-in-picture
minimap rendered to its own offscreen framebuffer (render-to-texture).
**Builds on:** v0.4 UBO + HDR framebuffer (merged to `main`).

## 1. Goal & Motivation

In v0.4 the offscreen-then-composite flow is hardcoded inside `OpenGLBackend`
(`beginFrame` binds the HDR FBO, `endFrame` composites). That worked for one
scene target but does not compose. v0.5 makes rendering **pass-oriented**:

1. **`RenderPass`** — a small value (`target` framebuffer + clear settings). The
   hardcoded two-pass flow becomes named, sequenced passes.
2. **Renderer-driven orchestration** — the backend becomes pass primitives
   (`beginPass`, `execute`, `blit`); the `Renderer` owns the framebuffers and
   sequences passes.
3. **Render-to-texture** — a second offscreen framebuffer holds a top-down
   minimap of the swarm, composited as a picture-in-picture inset. This proves
   `RenderPass`/`Framebuffer` support multiple targets and cameras, and is the
   seam future shadow maps, mirrors, portals, and debug views build on.

Scope is disciplined: no render graph, no material-system changes, no UI
framework, no game-facing pass API. Just `RenderPass`, a second `Framebuffer`,
the backend primitives, and explicit pass sequencing in the `Renderer`.

## 2. Roadmap context (this spec is v0.5 only)

| Milestone | Adds |
| --- | --- |
| v0.1–v0.4 (done) | Command buffer, sort keys, mesh path, JobSystem, camera UBO, HDR framebuffer + tonemap. |
| **v0.5 (this spec)** | `RenderPass` value, Renderer-driven pass sequencing, backend `beginPass`/`blit` primitives, PiP minimap render-to-texture. |
| Later | Game-facing pass API / render graph, MRT G-buffer + deferred lighting, bloom/exposure, shadow maps, GPU instancing, Vulkan backend. |

## 3. Decisions (locked)

| Decision | Choice |
| --- | --- |
| RenderPass | Plain value `struct RenderPass { Ref<Framebuffer> target; glm::vec4 clearColor; bool clearDepth; }` (null target = default framebuffer) |
| Orchestration | Moves from the backend to the `Renderer`; backend gains `beginPass`/`blit`, loses `beginFrame`/`endFrame` |
| Scene FBO owner | Moves from backend to `Renderer` |
| Minimap | Existing top-down camera, fixed **512×512** HDR framebuffer, inset top-right |
| Blit | Tonemapped (Reinhard + gamma) quad with a `u_rect` NDC uniform; used fullscreen and for the inset |
| Pass API | Internal to the `Renderer` for v0.5 (no game-facing pass API) |
| Command reuse | The single merged command list is executed twice (two cameras / two targets); no re-generation |
| Depth ordering | Sort-key depth stays main-camera-relative; the minimap relies on the depth test |
| DoD | Tonemapped fullscreen swarm + a square top-down minimap inset in the top-right corner |

## 4. Architecture & data flow

```
BotArenaGame::onRender
  │  renderer.setCamera(*mainCamera)
  │  renderer.setMinimapCamera(m_topDownCamera)   // fixed square top-down view
  │  serial grid + walls; parallel swarm (unchanged)
  ▼
Renderer::beginFrame(w,h): reset lanes; m_sceneFBO->resize(w,h); store w,h
  (layers submit into per-lane queues)
Renderer::endFrame
  │  merge lanes + std::stable_sort -> m_merged
  │  // pass 1: minimap (render-to-texture)
  │  backend.beginPass(m_minimapFBO.get(), minimapClear, true, 512, 512)
  │  backend.execute(m_merged, m_minimapCamera, arena, registry)
  │  // pass 2: scene (HDR)
  │  backend.beginPass(m_sceneFBO.get(), sceneClear, true, w, h)
  │  backend.execute(m_merged, m_camera, arena, registry)
  │  // pass 3: composite (default framebuffer)
  │  backend.beginPass(nullptr, black, false, w, h)
  │  backend.blit(m_sceneFBO->colorAttachment(),   {-1,-1, 1, 1})      // fullscreen
  │  backend.blit(m_minimapFBO->colorAttachment(), minimapRect(w,h))   // inset
  ▼
saveScreenshot -> readPixels(default framebuffer)   // composited image
```

Only command generation is unchanged. All GL stays in the backend; the
`Renderer` sequences passes through the backend primitives and the `Framebuffer`
abstraction, so a future Vulkan backend slots in unchanged.

## 5. RenderPass

`engine/renderer/RenderPass.hpp`:

```cpp
#include <glm/glm.hpp>
#include "engine/core/Base.hpp"
#include "engine/renderer/Framebuffer.hpp"

struct RenderPass {
  Ref<Framebuffer> target;   // null == default framebuffer
  glm::vec4 clearColor{0.08f, 0.09f, 0.11f, 1.0f};
  bool clearDepth = true;
};
```

Backend-agnostic data. The `Renderer` holds three of these (minimap, scene,
composite) and hands their fields to `backend.beginPass`.

## 6. RenderBackend primitives

Replace `beginFrame`/`endFrame` with pass primitives:

```cpp
class RenderBackend {
 public:
  virtual void beginPass(Framebuffer* target, const glm::vec4& clearColor,
                         bool clearDepth, int viewportW, int viewportH) = 0;
  virtual void execute(const std::vector<RenderEntry>&, const CameraUniforms&,
                       Arena&, const ResourceRegistry&) = 0;
  virtual void blit(uint32_t sourceColorTexture, const glm::vec4& dstRectNDC) = 0;
  virtual void readPixels(int x, int y, int width, int height, void* out) = 0;
  static Scope<RenderBackend> Create();
};
```

`OpenGLBackend`:
- **`beginPass`**: if `target`, `target->bind()` (sets the FBO + its own viewport);
  else `glBindFramebuffer(GL_FRAMEBUFFER, 0)` + `glViewport(0,0,viewportW,
  viewportH)`. Then `glClearColor(clearColor)`; `glClear(COLOR |
  (clearDepth ? DEPTH : 0))`. Enables depth test when `clearDepth` (scene/minimap),
  leaves it as-is otherwise.
- **`execute`**: unchanged from v0.4 (uploads the camera UBO, draws the mesh +
  line passes into the currently bound target).
- **`blit(srcTexture, dstRectNDC)`**: `glDisable(GL_DEPTH_TEST)`; bind the blit
  program; `glBindTextureUnit(0, srcTexture)`; set `u_scene = 0` and
  `u_rect = dstRectNDC`; draw the unit quad; the shader maps the quad to
  `dstRectNDC` and Reinhard-tonemaps + gamma-corrects the sampled colour.
- Owns: line shader/VAO/VBO, camera UBO, the blit shader + a unit-quad VAO/VBO.
  **No longer owns the scene framebuffer** (moves to the `Renderer`).

Blit shader (inline): vertex maps a `[0,1]²` quad into `dstRectNDC`
(`gl_Position = mix(rect.xy, rect.zw, a_uv)` in NDC) and passes `a_uv`; fragment
samples `u_scene`, `c/(c+1)`, `pow(c, 1/2.2)`.

## 7. Renderer

- Owns `Ref<Framebuffer> m_sceneFBO` (HDR, `resize`d to the window each frame) and
  `Ref<Framebuffer> m_minimapFBO` (HDR, fixed 512×512), plus three `RenderPass`
  values and two `CameraUniforms` (`m_camera`, `m_minimapCamera`).
- `setCamera(const Camera&)` (exists). New
  `setMinimapCamera(const Camera& camera)` → `m_minimapCamera =
  makeCameraUniforms(camera.view(), camera.projection())`.
- `beginFrame(w,h)`: reset lanes; `m_sceneFBO->resize(w,h)`; store `w,h`. (No GL
  clear — passes clear.)
- `endFrame()`: merge + `std::stable_sort`; then the three passes from §4 using
  `m_backend->beginPass/execute/blit`.
- `saveScreenshot` unchanged (reads the composited default framebuffer).
- `generateMeshes`, per-lane buffers, `queue()`, `registry()`, `unitCubeMesh()`,
  `meshShader()` unchanged.

## 8. Minimap inset helper

`engine/renderer/MinimapRect.hpp` (GL-free, unit-testable):

```cpp
// Top-right inset NDC rect {x0,y0,x1,y1}; height is `heightFrac` of the screen,
// width compensates for aspect so a square source stays square on screen.
inline glm::vec4 minimapRect(int screenW, int screenH, float heightFrac,
                             float margin) {
  const float aspect = static_cast<float>(screenW) / static_cast<float>(screenH);
  const float hNdc = heightFrac * 2.0f;            // fraction -> NDC height
  const float wNdc = hNdc / aspect;                // keep square on screen
  const float x1 = 1.0f - margin;
  const float y1 = 1.0f - margin;
  const float x0 = x1 - wNdc;
  const float y0 = y1 - hNdc;
  return {x0, y0, x1, y1};
}
```

Called with `heightFrac = 0.32`, `margin = 0.02` (top-right, ~32% of screen
height, square).

## 9. Game

`BotArenaGame::onRender`: command generation unchanged. Before the passes run it
calls both `renderer.setCamera(*camera)` (active mode) and
`renderer.setMinimapCamera(m_topDownCamera)`. The top-down camera is configured
to a fixed square view covering the arena (e.g. `setBounds(-12, 12, -12, 12,
-100, 100)`), independent of the window aspect, so the minimap shows the whole
arena square. The `m_uiCamera` member (unused since v0.1) is left as-is.

## 10. Testing

- **Unit (Catch2, no GL):** `minimapRect` — for a 16:9 screen (`1280×720`,
  `heightFrac=0.32`, `margin=0.02`), the returned rect sits inside `[-1,1]`, its
  top-right corner is at `(0.98, 0.98)`, and its NDC width:height ratio equals
  `height/width` of the screen (i.e. the on-screen box is square:
  `(x1-x0)*screenW == (y1-y0)*screenH`).
- **Behavioral:** `BOTARENA_SCREENSHOT` shows the tonemapped swarm fullscreen
  **plus** a square top-down minimap of the swarm inset in the top-right corner —
  proving three passes, two cameras, and two render targets.

## 11. Out of scope (later milestones)

A game-facing pass API / render graph, MRT G-buffer + deferred lighting,
bloom / exposure / other post, shadow maps, mirrors/portals, GPU instancing, and
a second backend. The `RenderPass` value, the backend pass primitives, and the
per-target `Framebuffer` model are designed to accept these without changing
their surface.
