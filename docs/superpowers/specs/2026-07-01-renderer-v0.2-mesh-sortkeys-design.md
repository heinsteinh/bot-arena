# Renderer v0.2 — Mesh Path, RHI Resources & Sort Keys — Design Spec

**Date:** 2026-07-01
**Status:** Approved for planning
**Scope:** Engine v0.2 — add a filled-mesh render path on the command buffer,
introduce the first RHI resource abstractions (Buffer/VertexArray/Shader) with a
handle-based `ResourceRegistry`, and evolve the sort key to encode
shader/material/depth so draws batch and state changes collapse.
**Builds on:** v0.1 stateless command-buffer renderer (merged, `main`).

## 1. Goal & Motivation

v0.1 established the stateless command buffer: game code submits `RenderCommand`s
into a per-frame `Arena`-backed `RenderQueue`, which is sorted and executed by an
abstract `RenderBackend`. Every command was a wireframe line/cube and the sort
key was only `[layer:8][sequence:56]`.

v0.2 makes the two v0.1 seams *pay off*:

1. **Real sort keys.** Encode `shader` and `material` (and a coarse `depth`) into
   the key so that sorting groups draws that share GPU state. The backend then
   skips redundant shader/material binds — the state-change-minimization win that
   the whole architecture exists for.
2. **RHI resources.** Introduce `Buffer` (Vertex/Index), `VertexArray`, and
   `Shader` as `Ref<>`/`Create()` resources (the subset of the RHI-foundation
   design needed now), and render real filled geometry through them.

The arena walls and obstacle cubes become filled, flat-shaded meshes; the grid
and the bot-tracking line stay wireframe. The visible scene changes (filled
cubes), so the definition of done is a *new* screenshot, not an identical one.

## 2. Roadmap context (this spec is v0.2 only)

| Milestone | Adds |
| --- | --- |
| v0.1 (done) | Command buffer, layers, `[layer:8][sequence:56]` key, per-frame Arena, DebugRenderer, RenderBackend + OpenGLBackend. |
| **v0.2 (this spec)** | Filled-mesh path; `Buffer`/`VertexArray`/`Shader` RHI resources; handle-based `ResourceRegistry`; `[layer|shader|material|depth]` sort key + state-change-minimizing backend; `Material { baseColor }`; one hardcoded directional light. |
| v0.3 | JobSystem + thread-local arenas + parallel command generation → merge. |
| v0.4 | UniformBuffer, Texture, Framebuffer, RenderPass (offscreen/HDR). |
| v0.5 | Render graph. Later: Vulkan backend, GPU-driven culling. |

The earlier RHI-foundation spec (`2026-07-01-rhi-foundation-design.md`) remains
the reference for the resource abstractions; v0.2 implements its
Buffer/VertexArray/Shader subset. Its Texture/UniformBuffer/Framebuffer/
RenderPass land in later milestones.

## 3. Decisions (locked)

| Decision | Choice |
| --- | --- |
| Deliverable | RHI resources + filled-mesh path + material/shader sort keys |
| Resource refs in commands | Integer handles (`uint32`) into a `ResourceRegistry`; keeps commands POD |
| Shading | Flat-shaded; `Material { glm::vec4 baseColor }`; one hardcoded directional light + ambient |
| RHI resources in scope | `Buffer` (VertexBuffer + IndexBuffer), `VertexArray`, `Shader` |
| RHI resources deferred | `Texture`, `UniformBuffer`, `Framebuffer`, `RenderPass` |
| Sort key | `[ layer:8 | shader:16 | material:16 | depth:24 ]`; `std::stable_sort` for determinism |
| Camera matrix | `u_viewProjection` set per-shader as a plain uniform (no UBO yet) |
| Mesh submission | New `MeshRenderer` producer owns depth/sort-key computation |
| Wireframe `Cube` type | Kept available in `DebugRenderer`, unused by the scene |
| Resource model | `Ref<>` + `Create()` switching on a backend enum (OpenGL only) |
| DoD | Screenshot shows filled flat-shaded cubes (lighting gradient) + wireframe grid + yellow line |

   │  DebugRenderer debug(renderer.queue());

## 5. Sort key

`engine/renderer/RenderCommand.hpp`:

```cpp
// [ layer:8 | shader:16 | material:16 | depth:24 ]
inline uint64_t makeSortKey(RenderLayer layer, uint16_t shader,
                            uint16_t material, uint32_t depth24) {
  return (static_cast<uint64_t>(layer)    << 56) |
         (static_cast<uint64_t>(shader)   << 40) |
         (static_cast<uint64_t>(material) << 24) |
         (static_cast<uint64_t>(depth24 & 0x00FFFFFFu));
}
```

- Line/grid commands pass `shader=material=0, depth24=0`.
- Mesh commands pass the shader handle, material handle, and a 24-bit
  front-to-back depth (0 = nearest) computed by `MeshRenderer`.
- `RenderQueue::sort()` uses `std::stable_sort` (ascending key), so equal-key
  commands keep submission order — this replaces v0.1's sequence-in-key +
  `std::sort`. `RenderQueue` no longer needs its `m_sequence` counter.

Depth encoding uses only `Camera::view()` (available on every camera — no far-plane
getter needed): the view-space Z of the transform's translation, normalized by a
fixed `kFarPlane = 100.0f` constant:

```cpp
float viewZ = (camera.view() * glm::vec4(translation, 1.0f)).z;  // negative in front
float d = glm::clamp(-viewZ / kFarPlane, 0.0f, 1.0f);            // 0 = nearest
uint32_t depth24 = static_cast<uint32_t>(d * 0x00FFFFFF);
```

Coarse but sufficient to demonstrate front-to-back ordering within a material.

## 6. ResourceRegistry & Material

`engine/renderer/ResourceRegistry.hpp` (+ `.cpp`):

```cpp
struct Material {
  glm::vec4 baseColor{1.0f};
  uint16_t shader = 0;   // shader handle this material draws with
};

using MeshHandle = uint16_t;
using ShaderHandle = uint16_t;
using MaterialHandle = uint16_t;

class ResourceRegistry {
 public:
  ShaderHandle registerShader(const Ref<Shader>&);
  MeshHandle registerMesh(const Ref<VertexArray>&);
  MaterialHandle registerMaterial(const Material&);

  const Ref<Shader>& shader(ShaderHandle) const;
  const Ref<VertexArray>& mesh(MeshHandle) const;
  const Material& material(MaterialHandle) const;

 private:
  std::vector<Ref<Shader>> m_shaders;
  std::vector<Ref<VertexArray>> m_meshes;
  std::vector<Material> m_materials;
};
```

Handles are 16-bit vector indices (match the sort-key field widths). A material
carries the shader handle it draws with, so `MeshRenderer` derives the key's
`shader` field from the material.

## 7. RHI resources (Buffer / VertexArray / Shader)

Implemented per the RHI-foundation design, `Ref<>`/`Create()` switching on a
backend enum:

- **`Buffer.hpp`** — `enum class ShaderDataType`, `ShaderDataTypeSize`,
  `BufferElement`, `BufferLayout` (computes stride/offset; header-only,
  unit-tested); abstract `VertexBuffer` (`Create(size)` dynamic / `Create(data,
  size)` static, `setData`, `setLayout`, `layout`) and `IndexBuffer`
  (`Create(uint32*, count)`, `count`). `OpenGLBuffer.{hpp,cpp}` uses DSA
  (`glCreateBuffers`, `glNamedBufferData`).
- **`VertexArray.hpp`** — abstract (`addVertexBuffer`, `setIndexBuffer`,
  `indexBuffer`, `bind`). `OpenGLVertexArray` wires attributes from the layout
  via DSA (`glVertexArrayAttribFormat`).
- **`Shader.hpp`** — `ShaderSources`, static header-only `ParseSources` (splits
  `#type vertex`/`#type fragment`; unit-tested, no GL); abstract `Shader`
  (`Create(filepath)`, `Create(name, vert, frag)`, `bind`, `setInt/Float/
  Float3/Float4/Mat4`, `name`). `OpenGLShader.{hpp,cpp}` compiles/links, caches
  uniform locations.
- **`Base.hpp`** already provides `Ref`/`Scope`/`CreateRef`/`CreateScope` (v0.1).
- **Backend selection:** with only OpenGL present, each resource `Create()`
  constructs its `OpenGL*` implementation directly (e.g. `return
  CreateRef<OpenGLVertexArray>()`). A runtime backend-selector enum is deferred
  until a second backend (Vulkan) exists — the seam is a one-line switch to add
  then (YAGNI).
- **Shader assets on disk:** `assets/shaders/mesh.glsl` (`#type` sections). The
  line shader stays inline in `OpenGLBackend` (unchanged v0.1 path). Asset path
  resolved via a `BOTARENA_ASSET_DIR` compile definition + `assetPath(rel)`
  helper (`engine/core/AssetPath.hpp`).

## 8. Command & producers

- **`RenderCommand`** gains `RenderCommandType::Mesh` and fields used only by
  `Mesh`: `MeshHandle mesh; MaterialHandle material; glm::mat4 transform;` (the
  shader handle is looked up from the material at execute time and packed into
  the key at submit time). Still a flat POD; arena bump-copy unaffected.
- **`MeshRenderer`** (`engine/renderer/MeshRenderer.hpp` + `.cpp`) — holds
  `RenderQueue&`, `const ResourceRegistry&`, and `const Camera&` (uses only
  `camera.view()` for depth). `submit(MeshHandle, MaterialHandle, const
  glm::mat4& transform)`:
  1. reads `material.shader` from the registry for the key's shader field;
  2. computes `depth24` from the transform's translation via `camera.view()`
     (see §5);
  3. `makeSortKey(RenderLayer::Opaque, shader, material, depth24)`;
  4. builds and submits the `Mesh` command.
- **`DebugRenderer`** unchanged.

## 9. Backend

`RenderBackend::execute` signature gains the registry:
`execute(const std::vector<RenderEntry>&, const glm::mat4& viewProjection,
Arena& scratch, const ResourceRegistry& registry)`.

`OpenGLBackend`:
- **Line** commands → unchanged v0.1 dynamic-VBO batching (one `glDrawArrays`).
- **Mesh** commands → for each, resolve `registry.mesh/shader/material`:
  - if the shader handle differs from the currently bound one, `shader->bind()`
    and set `u_viewProjection` (once per shader run);
  - if the material handle differs, set `u_baseColor` (once per material run);
  - always set `u_transform`; bind the mesh `VertexArray`; `glDrawElements`.
  - `glEnable(GL_DEPTH_TEST)` and `glEnable(GL_CULL_FACE)` for the mesh pass;
    restore state for lines.
- Optional instrumentation: count shader/material binds and `glDrawElements`
  calls; `spdlog::debug` them once so batching is observable.
- The unit-cube mesh (position + normal, 24 vertices, 36 indices) is created and
  registered once at startup and reused by all filled cubes.

`assets/shaders/mesh.glsl`:

```glsl
#type vertex
#version 460 core
layout(location = 0) in vec3 a_position;
layout(location = 1) in vec3 a_normal;
uniform mat4 u_viewProjection;
uniform mat4 u_transform;
out vec3 v_normal;
void main() {
  v_normal = mat3(u_transform) * a_normal;
  gl_Position = u_viewProjection * u_transform * vec4(a_position, 1.0);
}

#type fragment
#version 460 core
in vec3 v_normal;
uniform vec4 u_baseColor;
out vec4 fragColor;
void main() {
  vec3 N = normalize(v_normal);
  vec3 L = normalize(vec3(0.4, 0.8, 0.3));
  float diffuse = max(dot(N, L), 0.0);
  float ambient = 0.25;
  fragColor = vec4(u_baseColor.rgb * (ambient + diffuse), u_baseColor.a);
}
```

## 10. Renderer integration

`Renderer` (v0.1) gains: it owns a `ResourceRegistry`; a `registry()` accessor
(mutable, so the game can register meshes/materials); `setViewProjection` already
exists (used by the backend for `u_viewProjection`); `endFrame` passes the
registry to `backend->execute`. Depth needs no `Renderer` camera state —
`MeshRenderer` holds the camera directly. The mesh shader + unit-cube mesh are
built and registered once after the GL context is current (the `Renderer`
constructor already runs then, per v0.1); their handles are exposed via
`registry()` or stored for the game to read.

`BotArenaGame`:
- On first render, obtains the cube mesh handle + a set of material handles (one
  per cube color) from the registry (created by the game or by a small helper).
- `onRender`: sets camera on the renderer; submits filled cubes via
  `MeshRenderer` (Opaque) and grid + bot-line via `DebugRenderer`.

## 11. Scene mapping

| Element | v0.1 | v0.2 |
| --- | --- | --- |
| Grid | `Line` / Grid | unchanged (wireframe) |
| Wall + obstacle + bot cubes | `Cube` wireframe / Opaque | **`Mesh` filled / Opaque** (cube mesh + per-color `Material`) |

## 12. Testing

- **Unit (Catch2, no GL):**
  - `makeSortKey`: layer in bits 56-63, shader 40-55, material 24-39, depth
    0-23; ordering layer > shader > material > depth; depth masked to 24 bits.
  - `RenderQueue::sort`: `std::stable_sort` orders across layers and, within
    equal keys, preserves submission order (line determinism); mesh commands with
    different shader/material/depth order correctly.
  - `ResourceRegistry`: register returns increasing handles; `get(handle)`
    returns the registered resource; handles stay valid as more are added.
  - `BufferLayout`: stride/offset for a position+normal layout.
  - `Shader::ParseSources`: splits `mesh.glsl`-style `#type` sections.
- **Behavioral:** `BOTARENA_SCREENSHOT` renders filled, flat-shaded cubes with a
  visible lighting gradient across faces, plus the wireframe grid and yellow
  line. Manually confirmed (the image differs from v0.1 by design — cubes filled).
- **Batching (optional instrumentation):** with all cubes sharing one shader and
  the cube mesh, backend logs should show a single shader bind and one
  `glDrawElements` per material run — evidence the sort key collapsed state.

## 13. Out of scope (later milestones)

`Texture`/`UniformBuffer`/`Framebuffer`/`RenderPass`, a real light/material
system (multiple lights, PBR), instanced/indirect drawing, job-system
multithreading + thread-local arenas, type-erased command packets, render graph,
Vulkan backend. The command-buffer API, the handle/registry model, and the sort
key are designed to accept these without changing the public surface.
